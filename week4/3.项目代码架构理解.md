# Week 4 项目代码架构理解

本文档用于帮助理解 Week4 工程的代码组织、运行流程和各模块之间的关系。

本项目主题是 **RAG and Agent Knowledge Assistant**，目标是用一个离线可运行的 Python 工程，演示 RAG、Agent 和 MCP-like 工具调用的基本工程结构。

---

## 1. 项目整体定位

Week4 项目不是 Notebook，而是一个 Python 程序工程。它更接近真实 AI 应用的组织方式：

- 文档需要从磁盘读取。
- 文本需要切分成 chunk。
- 检索索引需要保存到本地文件。
- 问答程序需要从命令行接收参数。
- Agent 需要根据用户问题选择工具。
- MCP-like 示例需要用结构化方式描述工具和调用工具。

本项目为了便于教学和离线运行，没有接入真实大模型 API，也没有使用向量数据库。它使用：

- `TF-IDF` 作为文本表示方式。
- `cosine_similarity` 作为检索相似度计算方式。
- 简单模板作为回答生成方式。
- 规则判断作为 Agent 的工具选择方式。

因此，它的重点不是回答质量，而是帮助理解一个 RAG/Agent 项目的最小工程闭环。

## 5. 文档加载模块：document_loader.py

`src/document_loader.py` 负责把本地知识库文档读取成文本，并切分成 chunk。

### 5.1 支持的文件类型

当前支持三类文件：

```python
supported_suffixes = {".md", ".txt", ".pdf"}
```

### 5.2 文本文件读取

`.md` 和 `.txt` 文件通过 UTF-8 文本方式读取：

```python
def read_text_file(path: Path) -> str:
    return path.read_text(encoding="utf-8")
```

### 5.3 PDF 文件读取

`.pdf` 文件通过 `pypdf.PdfReader` 逐页提取文本：

```python
def read_pdf_file(path: Path) -> str:
    reader = PdfReader(str(path))
    pages = []
    for page in reader.pages:
        text = page.extract_text() or ""
        pages.append(text)
    return "\n".join(pages)
```

注意：`pypdf` 只能提取 PDF 中真实存在的文本。如果 PDF 是扫描图片，需要额外使用 OCR。

### 5.4 加载文档

`load_documents(docs_dir)` 会递归扫描 `data/docs/`，读取所有支持类型的文件，并返回如下结构：

```python
{
    "source": "docs/kicad_basics.md",
    "text": "文档正文..."
}
```

其中：

- `source` 表示文档来源路径。
- `text` 表示文档完整文本。

### 5.5 文本切分

`chunk_text()` 负责把长文本切成多个 chunk：

```python
def chunk_text(text: str, chunk_size: int = 450, overlap: int = 80):
```

默认参数含义：

- `chunk_size=450`：每个 chunk 最多约 450 个字符。
- `overlap=80`：相邻 chunk 之间保留 80 个字符重叠。

重叠的作用是减少上下文被切断的问题。例如某个知识点刚好跨越两个 chunk，overlap 可以让后一个 chunk 仍然保留部分前文。

## 6. 索引构建模块：build_index.py

`src/build_index.py` 负责把文档转换为可检索索引。

它的核心流程是：

```text
确保目录存在
-> 加载 data/docs/ 中的文档
-> 对每篇文档切分 chunk
-> 生成 chunks DataFrame
-> 使用 TfidfVectorizer 建立文本向量
-> 保存索引到 rag_index.pkl
-> 保存 chunk 明细到 chunks.csv
```

### 6.1 构建 chunk 表

每个 chunk 会被整理成一行：

```python
{
    "source": document["source"],
    "chunk_id": chunk_id,
    "text": chunk,
}
```

最终生成 `chunks_df`，包含：

- `source`：chunk 来自哪篇文档。
- `chunk_id`：该 chunk 在文档中的编号。
- `text`：chunk 文本。

### 6.2 TF-IDF 向量化

项目使用 scikit-learn 的 `TfidfVectorizer`：

```python
vectorizer = TfidfVectorizer()
matrix = vectorizer.fit_transform(chunks_df["text"].tolist())
```

含义：

- `vectorizer` 学习所有 chunk 中出现的词汇和权重。
- `matrix` 是所有 chunk 的 TF-IDF 表示。

### 6.3 保存索引

索引保存为一个 pickle 文件：

```python
payload = {
    "vectorizer": vectorizer,
    "matrix": matrix,
    "chunks": chunks_df.to_dict(orient="records"),
}
```

保存位置：

```text
vector_store/rag_index.pkl
```

同时，chunk 明细也会保存为 CSV：

```text
vector_store/chunks.csv
```

`rag_index.pkl` 给程序读取使用，`chunks.csv` 给人查看和调试使用。

## 7. RAG 问答模块：rag_ask.py

`src/rag_ask.py` 负责完成一次 RAG 问答。

它包含三个关键步骤：

```text
加载索引
-> 检索相关 chunk
-> 根据检索结果生成回答
```

### 7.1 加载索引

`load_index()` 会读取：

```text
vector_store/rag_index.pkl
```

如果索引不存在，会提示先运行：

```powershell
python src/build_index.py
```

### 7.2 检索相关内容

`retrieve(question, top_k=3)` 是检索核心。

流程如下：

```text
用户问题
-> vectorizer.transform([question])
-> 得到问题 TF-IDF 向量
-> 与所有 chunk 向量计算 cosine similarity
-> 按分数从高到低排序
-> 返回 top_k 个 chunk
```

代码中的相似度计算：

```python
scores = cosine_similarity(query_vec, matrix).flatten()
top_indices = scores.argsort()[::-1][:top_k]
```

返回结果中包含：

- `source`
- `chunk_id`
- `text`
- `rank`
- `score`

### 7.3 生成回答

`build_answer(question, contexts)` 目前使用的是简单模板。

它会把检索到的第一条 chunk 作为主要回答内容，并附上参考来源：

```text
问题：...

基于知识库检索到的内容，参考答案如下：

第一条检索结果文本

参考来源：
检索结果列表
```

这一步在真实 RAG 系统中通常会交给 LLM 完成，例如把多个检索结果作为上下文传给大模型，让模型综合生成答案。

### 7.4 保存结果

每次问答后，会写入：

```text
results/last_rag_answer.md
results/last_retrieval_results.csv
```

其中：

- `last_rag_answer.md` 保存最终回答。
- `last_retrieval_results.csv` 保存检索命中的 chunk、分数和来源。

这对于理解 RAG 很重要，因为可以看到“答案为什么会这样生成”。

## 8. 工具模块：tools.py

`src/tools.py` 定义了 Agent 可以调用的工具。

当前工具列表保存在 `TOOLS` 字典中：

```python
TOOLS = {
    "list_documents": {
        "description": "List all documents in the local knowledge base.",
        "function": list_documents,
    },
    "search_knowledge": {
        "description": "Search the local RAG knowledge base and return an answer.",
        "function": search_knowledge,
    },
    "calculate": {
        "description": "Evaluate a simple numeric expression.",
        "function": calculate,
    },
}
```

每个工具包含：

- `description`：工具说明，告诉 Agent 这个工具能做什么。
- `function`：实际要执行的 Python 函数。

### 8.1 list_documents

列出 `data/docs/` 中的知识库文档：

```python
def list_documents():
```

返回示例：

```text
kicad_basics.md
pcb_design_flow.md
rag_agent_notes.md
```

### 8.2 search_knowledge

调用 RAG 问答：

```python
def search_knowledge(question: str):
    return ask(question, top_k=3)
```

它本质上是 `rag_ask.py` 的工具封装。

### 8.3 calculate

计算简单数学表达式：

```python
def calculate(expression: str):
```

它使用 Python 的 `ast` 模块解析表达式，并只允许部分安全的数学操作：

- 加法
- 减法
- 乘法
- 除法
- 幂运算
- 负号

这种方式比直接使用 `eval()` 更安全。

## 9. Agent 示例：agent_demo.py

`src/agent_demo.py` 展示了一个最小 Agent 的工作方式。

它的核心思想是：

```text
用户输入问题
-> Agent 判断应该使用哪个工具
-> 调用工具
-> 返回工具结果
```

### 9.1 自动构建索引

`run_agent()` 会先检查索引是否存在：

```python
if not INDEX_PATH.exists():
    print("Index not found. Building index first...")
    build_index()
```

如果没有索引，会自动调用 `build_index()`。

### 9.2 工具选择逻辑

`choose_tool(question)` 使用简单规则选择工具：

```text
如果问题包含“列出”、list 或 documents
-> 选择 list_documents

如果问题包含“计算”或 calculate
-> 选择 calculate

否则
-> 选择 search_knowledge
```

示例：

```powershell
python src/agent_demo.py --question "请列出知识库文档"
```

会选择：

```text
list_documents
```

示例：

```powershell
python src/agent_demo.py --question "计算 12 * 8 + 4"
```

会选择：

```text
calculate
```

示例：

```powershell
python src/agent_demo.py --question "AI 可以如何辅助 PCB 设计？"
```

会选择：

```text
search_knowledge
```

### 9.3 Agent 与普通脚本的区别

普通脚本通常是固定流程，例如“输入问题 -> 检索 -> 回答”。

Agent 多了一层决策：

```text
我应该调用哪个工具？
这个工具需要什么参数？
调用结果如何返回给用户？
```

本项目的 Agent 决策规则很简单，但结构上已经具备 Agent 的基本形态。

## 10. MCP-like 示例：mcp_like_demo.py

`src/mcp_like_demo.py` 用简单 JSON 模拟 MCP 的核心思想。

MCP 的重点是：

```text
模型或客户端可以用标准方式发现工具、理解工具、调用工具、获得结果。
```

本项目没有启动真正的 MCP Server，而是用普通 Python 函数模拟。

### 10.1 工具发现

`describe_tools()` 会返回所有工具的名称和描述：

```json
[
  {
    "name": "list_documents",
    "description": "List all documents in the local knowledge base."
  },
  {
    "name": "search_knowledge",
    "description": "Search the local RAG knowledge base and return an answer."
  },
  {
    "name": "calculate",
    "description": "Evaluate a simple numeric expression."
  }
]
```

这对应 MCP 中的“工具发现”。

### 10.2 工具调用

`call_tool(tool_name, arguments)` 根据工具名调用实际函数：

```python
def call_tool(tool_name: str, arguments: dict):
    if tool_name not in TOOLS:
        raise ValueError(f"Unknown tool: {tool_name}")
    return TOOLS[tool_name]["function"](**arguments)
```

调用请求示例：

```json
{
  "tool_name": "list_documents",
  "arguments": {}
}
```

这对应 MCP 中的“工具调用”。

## 15. 当前实现的局限

这个项目为了教学简单，有一些有意保留的简化：

### 15.1 没有真实大模型

回答生成只是模板拼接，不会真正综合、推理或改写。

真实 RAG 中通常会把检索结果传给 LLM，由 LLM 生成自然语言答案。

### 15.2 检索方式比较基础

当前使用 TF-IDF，适合演示关键词匹配，但语义理解能力有限。

例如同义词、复杂中文语义、跨语言表达，TF-IDF 都不如 embedding 检索稳定。

### 15.3 Agent 决策是规则判断

当前 Agent 通过关键字判断工具：

```text
列出 -> list_documents
计算 -> calculate
其他 -> search_knowledge
```

真实 Agent 通常会由 LLM 根据工具描述和用户意图选择工具。

### 15.4 MCP-like 不是完整 MCP Server

当前示例只是模拟工具发现和工具调用，没有实现完整 MCP 协议、传输层、schema 校验或服务端生命周期。

## 16. 后续可扩展方向

可以从以下方向继续升级：

### 16.1 接入 Embedding 模型

用 embedding 替换 TF-IDF，例如：

```text
chunk text -> embedding vector -> vector search
```

这样可以提升语义检索能力。

### 16.2 使用向量数据库

可以把 `rag_index.pkl` 替换为：

- FAISS
- Chroma
- Milvus
- Qdrant

这样更接近真实生产系统。

### 16.3 接入真实 LLM

可以把 `build_answer()` 替换为 LLM 调用：

```text
question + retrieved contexts -> LLM -> final answer
```

### 16.4 增强 Agent 工具选择

可以用 LLM 根据 `TOOLS` 中的 description 自动选择工具，而不是用固定规则。

### 16.5 实现真正 MCP Server

可以把 `TOOLS` 暴露为真正的 MCP tools，让外部客户端通过 MCP 协议发现和调用。

### 16.6 改进中文处理

当前 TF-IDF 默认分词方式对中文不够理想。可以加入：

- `jieba` 中文分词。
- 自定义 tokenizer。
- 中文 embedding 模型。

## 17. 一句话总结

这个 Week4 项目是一个最小但完整的 RAG/Agent 教学工程：

```text
它用本地文档构建检索索引，用检索结果生成问答，用工具字典组织 Agent 能力，并用 JSON 形式模拟 MCP 的工具发现与调用。
```

理解这个项目之后，就可以进一步把其中的简化模块逐步替换成真实工程组件，例如 embedding、向量数据库、LLM API、LangChain、LlamaIndex 或真正的 MCP Server。
